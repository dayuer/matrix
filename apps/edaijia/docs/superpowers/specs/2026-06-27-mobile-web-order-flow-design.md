# 移动端 Web 叫代驾下单链路 · 设计文档

> 日期：2026-06-27　状态：设计已确认（含两轮场景审定 + 一轮审计），待评审
> 目标：把搜索引擎来的移动端流量，在 Web 上直接接住，走完一条「说需求 → AI 解析 → 下单一刻验证手机号 → 进队列人工审核 → 状态时间线跟踪」的下单链路。全程无独立注册——注册动作溶解进下单本身。

---

## 0. 核心人群与设计约束

**核心用户 = 酒后用户**：操作能力弱、打字困难、口齿/方言影响语音识别、耐心低。整套链路据此约束：
- **少步骤、大按钮、强容错**；能定位就别让他打字，能一键就别让他选。
- **语音优先**，识别不准由出行助手 AI 追问 / 人工兜底，不报错卡死。
- **回头客"老地方一键复下单"**，把重度醉酒场景压到一两次点击。

## 1. 定位与边界

- **Web 只做"叫车下单"前半段（B 方案）**：用户填完需求 + 完成手机号验证 = 下单成功，订单进队列，由调度后台 / 人工审核承接真正派单。**派单、司机匹配在后台，不在 Web**。
- **但 Web 展示派单进度**：确认页即状态跟踪页，靠查单接口刷新 / 轮询，**只读呈现后台推进的状态时间线**（§4「下单之后」）。
- **一期只做立即单**：不做预约单（识别到未来时间意图 → 告知暂只支持立即叫车 + 出口）。
- **"下单成功"的定义**：拿到一份"需求结构化 + 手机号已验证"的有效订单并落入队列。
- **支付 / 结算**：Web **不收款**，代驾费用**线下直接支付给司机**，计费与结算在履约端完成。
- **AI 的职责**：听懂自然语言需求并结构化；**人工审核兜底**理解成本，AI 没听懂绝不卡死真实客户。
- **一期不做**：预约单、精确计价、精确 ETA、地图选点、Web 自助改终点、回头客首单历史地址、远程登出全部设备。详见 §10。

## 2. 设备分流（进站第 0 步，SSR 判定）

| 来源 | 首屏 |
|---|---|
| **移动端**（含搜索流量） | 首屏即"叫代驾"下单入口，营销内容在下方 |
| **PC** | 现有营销站；下单位置替换为「手机扫码下单」二维码 + 「打开 e代驾 App」，**绝不出现填单框** |

- 判定：UA + 视口兜底；服务端 SSR 直接渲染两套首屏，不靠前端闪烁切换（SEO 友好、搜索快照干净）。
- 下单链路（§4）**仅在移动端存在**。

## 3. 区域识别层（移动端，进站静默，喂给 AI 解析）

目的：把 POI 检索域从"全国"缩到"一座城"，消除地名重名，提升解析准确率与置信度阈值。三级策略，原则"0 打扰先有结果，要精度时才要授权"：

1. **IP 静默锁"推定城市"**：落地即按 IP（本地 IP 库 / CDN 地理头）判一个城市，AI 第一秒起就被约束在城内。无授权、无弹窗、不拖慢首屏。
2. **按需 GPS 精确起点**：用户点输入框 / 点「使用当前位置」时才请求一次 `navigator.geolocation`（需 HTTPS）。拿到坐标 → 调 `/api/geo`（逆地理编码，§7）→ 地址 + 周边 POI 候选。拒绝授权不卡，退回 IP 城市 + 手打起点。
3. **顶部常驻「当前城市 ▾」可一键切换**：识别错（VPN / 运营商网关跨城 / 出差）时用户一秒切城；**切城即重置 POI 检索域并重解析**。

**反哺解析规则**：
- 起点：优先 GPS 反查 + 周边 POI，城市锁兜底。
- 终点：城内检索收敛重名。
- **跨城例外**：AI 识别到"跨城 / 送回老家 / 到外地机场"意图时，**自动放开终点的城市范围**；城市锁只约束起点和解析上下文。

**服务范围校验**：识别城市后判断 e代驾 是否在该城 / 区开通（300+ 城市≠全国全覆盖）。**未开通 → 给"暂未开通 + 留资 / 推荐附近城市"出口，不放进下单链路**。终点超运营半径的细粒度情况，由下单后状态 + 主动消息兜底（§4「无司机 / 超范围」），一期不做地理围栏。

## 4. 核心链路（移动端新客）

**最简链路一句话**：**你在哪 → 要去哪 → 验证手机号 → 结束。** 下单只认**两要素**（起点、终点），**默认即时出发**（一期只做立即单，时间不作字段），其余全为可选、不打断。

**交互模型 ｜「出行助手 AI」IM 对话式（贯穿 Step 1–2）**
下单不是填表，是和**出行助手 AI**的一段对话流（chat thread）。用户气泡（打字或语音转写）与助手回复交替出现，**助手的"回复"是结构化卡片，不是闲聊**：
- 用户说 / 打一段（"我在国贸喝多了想叫代驾回回龙观"，语音或文字）→ 助手解析 → 在对话流里回一张**需求卡片**（已识别项亮显）。
- **识别不全 / 不确定的逐项补问**——一次只问一项，以助手气泡呈现（"终点具体到哪？"），用户可自然语言回、也可点快捷选项答；答完卡片就地更新。
- 卡片**任意字段可点改**；用户也能随时再补一句话覆盖（"改成望京SOHO"）。

**Step 1 ｜开场（进入即对话）**
首屏**直接就是对话界面**：顶部出行助手 AI 的招呼气泡（新客"去哪儿？说一句话我帮你叫代驾"／老客户"欢迎回来…"），**底部固定一个 voice-first 输入栏**——醒目的圆形麦克风主按钮 + 次要的"点这里打字"。**无 hero、无场景气泡、无登录、无表单字段**；右上角仅一个轻量「老客户登录」/ 已登录尾号入口，顶部「当前城市 ▾」。用户说 / 打第一句即进入解析（Step 2）。

**Step 2 ｜AI 解析 → 卡片 + 逐项补问（只认两要素）**
助手每收到一段都先做 **意图分流 + 缺项判断**，再谈 POI：
- **非"叫代驾"意图**（代检车 / 拖车 / 洗车 …）→ 对话里直接给出口（跳服务页 / 转客服），不进下单卡片。
- 是叫代驾 → 卡片核心 = **两要素**：`你在哪（起点） / 要去哪（终点）`。
- **逐项补问只补这两项**，缺哪问哪、一次一项；两项齐即可下单。
- **默认即时出发，时间不作字段**：一期只做立即单，卡片不显示时间行；**识别到未来时间 / 预约意图（"明天""6点""下周"）→ 告知一期只支持现在叫车**，给出口（转人工 / 二期预约），不进下单。
- **起点保守、终点激进**：起点**必须用户显式确认 + 可微调**（防 GPS 漂移导致司机扑空），哪怕 GPS 很准也让用户瞄一眼；终点可自动填。
- **可选、不追问**（折叠在"更多"，按需填）：乘车人手机号（默认 = 验证号，可改填他人，覆盖"替朋友叫"，司机优先联系乘车人）；备注 / 特殊要求（助手从原话自动带，如"酒后"）。**车型对代驾非必要，不作字段。**

**Step 3 ｜预估 → 「确认叫代驾」**
确认卡片下方给出**参考价格区间**（一期不接地图、不承诺具体到达分钟）。让用户在掏手机号前就拿到价格的确定感。一个大按钮：**确认叫代驾**。

**Step 4 ｜下单一刻 · 手机号验证（唯一卡点）**
点「确认叫代驾」后才验证，两条路径：
- **能命中本机号 → 一键登录**：「本机号 138\*\*\*\*8888 一键下单」一点完成，0 输入 0 等待，**同页无缝落单**（主路径）。
- **命中不了**（WiFi / 虚商 / PC 无 SIM / 探测失败）→ **验证短信链接（magic link）兜底**：用户输手机号 → 收一条短信，**点短信里的链接即完成验证 + 下单，无需输验证码**。link token 里签入本单订单要素，点开即在该环境渲染状态页（绕开 App 内置浏览器跳系统浏览器、cookie 不通丢会话的问题）。
  - **发链接后原页面进入"等待点击"态**：显示"验证短信已发出，点击短信里的链接完成下单"+「重新发送」，用户不困惑。
  - **短信兜底**：重发上限内仍未点击 / 未收到 → 降级「**留号 + 人工回拨下单**」（落单标记"待人工核实号码"，不阻塞），守住最后一米。
- 验证通过 = 下单成功 = 账号创建（号码即账号，用户无"注册"感知）。
- 验证前一行小字给《服务协议》《隐私政策》（默认勾选、可点开）。

> **验证短信链接一链三用**：① 验证手机号；② 持续触达（关页后点链接即回到订单状态页）；③ 查 / 取消的身份凭证（link token 即鉴权，堵住"凭手机号无密码即可查到你家地址"的隐私漏洞）。

**Step 5（对话兜底）｜AI 听不懂 / 信息太乱 / 解析失败**
- 置信度低不报错，在对话流里温和追问补齐。
- **同一必需项最多追问 2 轮**仍拿不到 → 给两条逃生门：(a) 带原话直接下单、人工补全；(b) 一键转人工 / 拨 95139。绝不在对话里无限打转。
- **解析超时 / AI 不可用**（弱网）：降级为保留原话直接下单（人工补全）或转人工 / 拨 95139。

**下单之后 ｜订单状态页（可刷新 + 时间线跟踪）**
- 确认页即**状态跟踪页**，**可刷新 / 轮询** `GET /api/order/:token` 拉最新，竖向**时间线**展示后台推进：
  `已受理 → 人工已确认 → 正在呼叫司机 → 司机已接单`（每步带时间戳）。
- **改终点**：Web 不做自助改——页面提示"如需更改目的地，请上车后直接告知司机"。
- **取消**：**司机接单前**可页面**自助点取消**（凭 link / 登录凭证）；司机已接单后 → 引导联系司机 / 拨 95139。
- **无司机 / 超出服务范围**：时间线显示终态 +**主动下发短信告知客户**（复用短信渠道），并给"改地址重试 / 转人工"出口。
- 身份：凭验证短信链接（或持久登录态）查 / 取消；无有效凭证只见"已受理"最简态、看不到地址（隐私保护）。
- 运营侧：订单 + AI 解析结果 + 用户原话 + 置信度，进**人工审核队列**；审核员看到已结构化、已标红疑点的卡片，可核 / 改 / 转真实派单系统，并回写状态驱动用户侧时间线。

## 5. 老客户识别与持久登录

主路径永远是**出行助手 AI**对话叫代驾，登录是次要、可选、不挡道。**所有文案走短。**

**尾号 xxxx 贯穿规则**：凡涉及"这单 / 这个账号是谁"的触点，统一用「尾号8888」指代当前手机号——确认身份、不暴露全号。落点：① 右上角已登录态「尾号8888 ▾」；② 老客户登录后顶部「欢迎回来，尾号8888」；③ 一键登录按钮「本机号 138\*\*\*\*8888 一键下单」；④ 状态页「司机将联系尾号8888」；⑤ 查 / 取消入口与改绑 step-up 处显示当前绑定尾号。（被动 localStorage chip 拿不到号码，仍只显示「👋 欢迎回来」，不带尾号。）

**两条识别通道**
- **被动（localStorage）**：认出老设备 → 首屏软提示 chip「👋 欢迎回来」（仅此一句，无推销尾巴）；常用地址气泡自然出现。
- **主动（右上角入口）**：未登录显示「老客户登录」→ 点开同一套验证（一键登录主 / 验证短信链接兜底）→ 拉历史 → 顶部「欢迎回来，尾号8888」+ 常用地址气泡。

**持久登录态**
- 首单 `/api/order` 验证成功时，**下发长期签名登录凭证**（localStorage + httpOnly cookie 双写）。
- 回访由 **Node BFF 自行校验签名**恢复登录态（0 次后台调用），仅拉最新历史 / 状态时才走 `GET /api/order/...`。
- **"永久"**：服务端不主动过期，每次回访滑动续期；终止只有 ① 用户登出；② 用户清缓存 / 换设备（退回一键登录，一点恢复）。

**回头客漏斗塌缩成 3 步**：进站 → BFF 校验凭证 → 已登录 → 对话 / 点常用地址 → 解析确认 → **确认叫代驾（直接下单，跳过验证）** → 状态页。掏手机号的唯一卡点对回头客消失。

**登录态 UI**：右上角未登录「老客户登录」/ 已登录「尾号8888 ▾」（含登出）。

## 6. 安全取舍

- **下单照常免验证**（尊重持久登录、零摩擦）。
- **改绑手机号触发 step-up 验证**（一键登录 / 验证短信链接）—— 已确认采用。
- 凭证与设备绑定 + 随时可登出兜底。
- 远程登出全部设备 → 二期。

## 7. 后端接口（Node BFF，核心后台压到最小）

打法：**Node 当 BFF**，前端只跟它同源对话；定位、地图 POI、AI、计价、运营商认证、短信、登录态签名全部编排 / 消化在 BFF 内，不外溢成"后台接口"。

| 接口 | 何时调 | BFF 内部编排 | 落核心后台 |
|---|---|---|---|
| `POST /api/parse` | 对话流里用户每发一段 | LLM 解析 + 意图分流 + 城内 POI 消歧 + 参考价区间，**一次返回**卡片 + 缺项 + 追问项 + 置信度。多轮携带上下文，BFF 无状态，不新增接口 | ❌ |
| `POST /api/geo` | GPS 精确起点 / 字段点改选点 / parse 内部消歧 | `type:reverse`（坐标→地址+周边 POI）或 `type:search`（关键字+城市→POI 候选）。调地图 provider（一期高德，provider 抽象在 BFF），**缓存热门 POI + 限频** | ❌ |
| `POST /api/verify/send` | 一键登录不可用时 | 发验证短信（含 magic link，token 签入本单订单要素）；含重发上限，超限转"留号人工回拨" | ❌ |
| `POST /api/order` | 一键登录点击 / magic-link 点击 / 留号兜底 | 身份已验证后落单进队列 + 下发登录凭证。**入参三选一**：① 一键登录 token ② 已点击的 magic-link token ③ 回头客持久登录凭证；留号兜底则标记"待人工核实" | ✅ 唯一写后台 |
| `GET /api/order/:token`（查状态时间线 / 取消） | 状态页刷新 · 轮询 · 取消 · 拉历史 | 凭 magic-link / 登录凭证鉴权；返回状态 + 时间线；取消为同资源写操作；无有效凭证只见"已受理"最简态 | ✅ 读写后台 |

- **真正动到 e代驾 核心后台的只有 2 个**：下单进队列、查 / 取消单（含状态回写）。其余在 BFF + 第三方内消化。
- 验证 + 建单合并进 `/api/order` 一次完成：少一个接口、少一份中间态、天然挡住未验证脏单（**未验证不创建订单**）。
- 状态时间线由后台回写、BFF 透传；登录态由 BFF 签名自校验，恢复登录态 0 次后台调用。

## 8. 防刷与合规

- **`/api/parse` 与 `/api/geo` 均按 IP / 会话限频 + 轮次 / 调用上限**，挡住刷对话烧 LLM、刷地图烧调用费。
- 一键登录 / 验证短信按号码 + IP 限频；同号短时重复下单做合并提示，保护短信成本与审核队列。
- 手机号验证前给协议勾选（默认勾选 + 可点开），满足合规不打断。

## 9. 转化埋点

- **4 个关键漏斗事件**：开场输入 → AI 出卡 → 确认叫代驾 → 验证成功，各埋曝光 / 进入 / 完成。
- 重点盯 **"确认叫代驾"→"验证成功" 流失**（掏手机号一刻）与**一键登录命中率**——两数决定漏斗成败。
- 另盯：回头客免验证下单占比、AI 解析置信度分布、兜底追问触发率、跨城单识别准确率、magic-link 点击率、**无司机率 / 超范围率**、状态页回访率。

## 10. 一期 / 二期边界

**一期**：设备分流、三级定位 + 服务范围校验、出行助手 AI（对话式）+ AI 解析（含意图分流 / 起终点策略 / 乘车人号 / 兜底逃生门）、**仅立即单**、最后一刻验证（一键登录主 + 验证短信链接兜底 + 留号人工回拨）、订单状态时间线 + 自助取消、老客户双通道 + 持久登录、改绑 step-up、参考价区间、5 个 BFF 接口（核心后台 2 个）。

**二期**：预约单、精确计价、精确 ETA、地图选点、Web 自助改终点、回头客首单历史地址预填、远程登出全部设备、更细的状态（服务中 / 完成）。

## 11. 已知取舍记录

- **一期只做立即单**：不做预约——预约漏单是误机误事级高风险，需独立提醒 / SLA，二期再做干净。识别到预约意图先给出口。
- **改终点交给司机**：下单后不在 Web 自助改目的地，直接告知司机，省一套改单状态机；自助改终点二期评估。
- **预估价取 B**：一期参考区间 + 以司机确认为准，省一个地图依赖；精确计价 / ETA 二期。
- **验证用 magic link 取代 OTP 码**：少了"记码 / 输码"（酒后用户尤其易错），且**天然挡假号 / 错号**（收不到链接就走不下去，订单质量比 OTP 更高）；代价是多一次"切短信点链接"、依赖短信送达，故配重发 + 留号人工回拨兜底。一键登录仍为主路径（0 切换）。link token 一链三用（验证 / 触达 / 查单鉴权）。
- **Web 展示状态但不派单**：确认页升级为只读状态时间线（刷新 / 轮询查单接口），派单仍在后台；让用户随时知道"到哪一步了"，降低酒后焦虑。
- **回头客首单历史地址**：验证在最后，`localStorage` / 未登录的老客户**首单**享受不到历史提速；登录或下单后下一单即被认出。可接受。
- **持久免验证下单**：拿到已登录设备即可下单产生费用，是体验丝滑的代价；以"下单零摩擦、改绑才验证 + 可登出"平衡。
